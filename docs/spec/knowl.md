# 知识云

## 📜 一句话定义

知识云是量潮系统的“语义中枢”：它通过本体建模，将碎片经验转化为可连接、可推理、可复用的认知资产。

🚫 设计禁令（Anti-Patterns）
- ❌ 不做“关键词标签云”（那是信息，不是知识）；  
- ❌ 不追求“全量知识入库”（只建模高频、高价值的核心概念）；  
- ✅ 只做“让关键概念在系统中拥有唯一身份和关系”。

有了知识云，量潮云就不再是一组工具的集合，而是一个具备共同语言、可自我演化的认知有机体。  
它让“拿走空瓶子”不只是一个比喻，而是一个可被系统理解、调用、组合的认知原语。

## 🧠 知识云的基本建模思想

1. 核心信念：知识 ≠ 信息，而是可推理的关系网络
知识的本质，不是孤立的事实（如“AI 能生成纪要”），  
而是概念之间的结构化关系（如“AI → 消除 → 计算性混乱 → 支持 → 执行云”）。  
本体（Ontology）就是这个关系网络的形式化表达。

2. 问题定义：组织最大的浪费，是“知道但无法连接”
- 团队积累了大量文档、决策、案例；
- 但它们彼此割裂，无法被跨场景复用；
- 结果：重复造轮子、决策无上下文、新人难融入。
知识云的目标，是让“已知”自动找到“需要它的地方”。

3. 人机分工模型
任务   人类负责   机器负责
定义核心概念   提出关键实体（如“混乱”“角色”“传播”）及其业务含义   —

构建本体骨架   约定顶层分类（如“混乱类型”“系统角色”“价值流阶段”）   提供可视化建模工具，支持 OWL/RDF 等标准

填充实例与关系   在写作/议事/执行中自然产出带语义的内容   自动抽取实体、识别关系、链接到本体节点

| 推理与推荐 | — | 基于本体推理：  
 • “这个新需求属于哪种混乱？”  
 • “类似决策曾出现在哪个项目？” |

✅ 人类定义“意义”，机器维护“连接”。

4. 最小可行能力：从文本到语义图谱
- 用户在任何子系统（如写作云）中输入内容；
- 系统自动：
  1. 识别提及的概念（如“思考者”“空瓶子”“传播责任”）；
  2. 映射到本体中的标准术语（避免“议事者” vs “议员”歧义）；
  3. 建立三元组（主体 - 谓词 - 客体），如：  
     （执行云）-（消除）-（计算性混乱）  
     （传播者）-（承担）-（声誉风险）
  4. 支持语义搜索与关联推荐（“查看所有关于‘价值性混乱’的讨论”）

5. 在量潮云中的角色定位
- 角色：Creator + Amplifier  
  - 作为 Creator：它产出的是可计算的知识资产（本体、关系、推理规则）；  
  - 作为 Amplifier：它让其他系统的输出（如议事结论、执行记录）获得长期语义价值。
- 契约：所有子系统在输出关键概念时，应尽量引用知识云中的标准术语（通过 ID 或 URI）。

6. 与其他“云”的协同
协同方   如何联动
议事云   议事中出现的“思考者”“行动者”自动链接到本体中的角色定义；辩论逻辑可沉淀为推理规则

写作云   写作时自动提示相关概念、过往案例；成文后反哺本体（新增实例或关系）

执行云   “拿走空瓶子”这类模式可抽象为“最小行动模式”本体类，供复用

沟通云   传播内容时，附带语义标签，确保接收方能准确理解上下文
