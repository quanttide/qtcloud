# 智能体云

## 本质重定义

Agent 是上下文的具身化（Embodiment of Context）。
它不是“一个会做事的 AI”，而是一段活的、可交互的上下文。

为什么“围绕上下文构建 Agent”更本质？

1. 功能是流变的，上下文才是稳定的
同一个 Agent（如“议长”），在不同场景下做的事完全不同：
在周会中：澄清议题、控制时间；
在写作中：提炼核心主张、检查逻辑一致性；
在冲突调解中：隔离情绪、聚焦事实。
变的是任务，不变的是它所承载的上下文角色（中立、结构化、裁决者）。
✅ Agent 的身份由上下文定义，而非由能力列表定义。

2. 上下文 = 角色 + 约束 + 目标 + 历史
一个真正的上下文包含：
角色契约（你是谁？思考者？传播者？）
行为约束（你能做什么？不能做什么？）
当前目标（要消除哪类混乱？）
对话历史（已达成哪些共识？遗留哪些冲突？）
Agent 必须内嵌这四层，才能避免“看似智能，实则胡说”。

3. 用户不管理 Agent，而是进入上下文
用户不需要说：“请调用写作 Agent”；
而是说：“我想写一篇关于混乱的文章” → 系统自动构建 写作上下文，并激活相应的 Agent 集（议长+思考者+行动者）。
Agent 是上下文的“器官”，不是独立的服务。

## 验收标准

最终，用户不应感知到 Agent 的存在。
他们感知到的是：
“当我陷入选择瘫痪，系统自动开启一个安全的自我对话空间”；
“当我转发内容，系统提醒我确认来源并绑定责任”；
“当我面对一堆杂乱笔记，系统问我：‘要不要先拿走一个空瓶子？’”
这些体验背后，是上下文在驱动 Agent，而 Agent 只是上下文的执行终端。
