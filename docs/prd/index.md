# QtCloud - 跨域协作目录

还记得主简介里的那句"销售云签单 → 事件发出 → 商务云收到，创建合同"吗？

这就是协作目录在做的事。

它不是又一个独立产品，而是藏在 QtCloud 背后的一层机制——让七个领域产品不需要人工对接就能自动协作。协作目录不参与业务逻辑，不存储业务数据，它只做一件事：**让每个产品知道其他产品发生了什么，以及自己该做什么**。

## 谁在用

两个角色，一个提供，一个消费：

| 角色 | 他们在协作目录里做什么 |
|------|----------------------|
| **域平台团队**（如销售云团队） | 对自己负责的领域产品声明："我发布这些事件，我订阅那些事件" |
| **业务流程设计者**（如你） | 在事件之间建立连接："当销售云签单后，商务云应该创建合同，资产云应该准备交付" |

## 核心功能

### 事件目录——每个产品在这里"自我介绍"

每个域平台在协作目录里声明自己发布什么、订阅什么。其他域能直接浏览和搜索，不用靠文档或口口相传。

```
销售云发布:
  - deal.created   新商机产生
  - deal.closed    商机成交（含金额、客户）
  - deal.updated   商机信息变更

销售云订阅:
  - customer.verified  来自客户数据域——客户资质通过了才推进

商务云发布:
  - contract.signed    合同已签署
  - contract.invoiced  合同已开票
```

看到 `deal.closed` 就知道：哦，销售云签单了会告诉我。看到 `contract.signed` 就知道：哦，商务云合同签好了我可以接棒了。

### 跨域流程模板——不用从零开始

最常见的跨域流程已经预置好，直接启用：

| 模板 | 故事线 |
|------|--------|
| 成交转化 | 销售把单签了 → 商务自动创建合同，不用销售再填一遍 |
| 订单交付 | 合同签了 → 资产自动准备交付，不用商务再通知一遍 |
| 开票收款 | 发票开了 → 财务自动生成应收，不用财务再录一遍 |

你也可以自定义——选一个发布事件，选一个订阅域，配置数据映射，保存即可生效。

### 契约版本管理——改之前先打招呼

销售云想改 `deal.closed` 的事件格式？协作目录会自动查出谁在订阅这个事件。

改之前，向所有订阅方发起变更通知。订阅方确认了，才能上线。这样销售云不会在不知情的情况下把商务云的流程搞断。

### 兼容性检测——改了不坏别人

订阅方可以定义自己对事件的期望——比如"我只要金额大于 0 的 deal"、"客户字段不能为空"。

销售云发布新版本时，协作目录自动用所有订阅方的期望去跑一遍。任何一个消费者不通过，变更就不能上线。

## 用户故事

**注册事件**

> 我是销售云团队的新人。老大说"去协作目录把咱们的 `deal.closed` 注册一下"。我登录协作目录，填了事件名称、字段列表、示例数据，保存。以后任何域都能搜到并订阅它。

**发现并订阅**

> 我是商务云团队的开发者。产品说"销售签单后我们要自动创建合同"。我打开协作目录，找到 `deal.closed` 事件，看了它的字段——有客户 ID、金额、产品列表——正好够用。点"订阅"，保存。这就完成了对接。

**编排跨域流程**

> 我是业务负责人。产品上线后我发现：销售签了合同，商务云和资产云应该协作，但没人定义这个流程。我在协作目录选"成交转化"模板，启用。三分钟的事。

**契约变更协调**

> 销售云团队要改 `deal.closed`，加一个新字段。提交后协作目录提示：有 2 个订阅方（商务云、资产云），需要他们确认。商务云回复"没问题，我们兼容易"，资产云回复"需要一周适配"。变更进入等待期，一周后自动上线。

## 协作目录不做什么

- 不传输事件——只管谁发了什么、谁该知道，不管消息怎么投递（那是事件总线的事）
- 不存业务数据——只看事件结构，不看事件内容（数据在各域的数据库里）
- 不做垂直功能——只做衔接，不管获客、不管报价、不管交付——那是各个领域产品的本事
